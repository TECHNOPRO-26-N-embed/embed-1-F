### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）─→ [計測中] ──（閾値超え）─→ [再入力待ち]
　↑　　　　　　　　　　　　　　　　　　 ｜
（リセットボタン）（いらないのが理想）  └→（閾値内）─→[音楽再生/停止]

```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| 待機中 | センサ検知するまで同じ状態 | 前回の状態をキープ |
| センサ検知 | 瞬間的な値が○○になる | ○○になったら音楽再生/停止 |
|  |  |  |

> [!TIP]

## 2. ソフトウェア設計

### 2-1. データ設計（使う変数・定数の洗い出し）

> ※ プログラムで扱う「情報の種類」を先に整理します。
> ソフトウェアワークシート（AI駆動開発）でやった「データの洗い出し」と同じです。

| 情報の名前 | 何を表すか | 型（イメージ） | バイト数 | 初期値 | 備考 |
|:--|:--|:--|:--|:--|:--|
| sensor | サウンドセンサの計測値 | int | 2 | 0 | A/D変換された音の大きさ |
| sound | 現在再生している曲の番号 | int | 2 | 0 | 0,1,2などで曲を管理 |
| playing | 音楽が再生中かどうか | bool | 1 | false | trueで再生中 |
| clapCount | 検出した拍手の回数 | int | 2 | 0 | 一定時間内の拍手回数 |
| firstClapTime | 最初の拍手の時刻 | unsigned long | 4 | 0 | millis()で管理 |
| noteIndex | 現在再生している音の位置 | int | 2 | 0 | 曲再生用 |
| lastNoteTime | 前回音を鳴らした時刻 | unsigned long | 4 | 0 | millis()で管理 |
| lastSound | 1つ前のセンサ値 | int | 2 | 0 | 拍手検出の精度向上 |
| lastTriggerTime | 最後に動作した時刻 | unsigned long | 4 | 0 | 入力無効制御 |

> | 合計バイト数 | 判定 |

### 2-2. 機能一覧・関数一覧（役割の定義）

> ※ プログラムを「処理の部品（関数）」に分割します。
> **ここではコードを書かず、「何をする関数か」だけ日本語で書いてください。**
> 具体的な実装は詳細設計書（detailed_design.md）で行います。

> [!TIP]
> **単一責任の原則（1関数 = 1つの仕事）**
> 「機能名」欄に requirements.md の機能名（必須・追加）を書き、それを実現する関数名を対応させてください。
> 入出力の読み取り（readButton 等）や制御（updateOutput 等）は複数機能から使われる共通処理として「共通」と書いてOKです。

| 機能ID | 機能名 | 関数名 | 担う「1つの仕事」 | 主な引数 | 戻り値 | 呼び出す場所 |
|:--|:--|:--|:--|:--|:--|:--|
| F1 | センサ読み取り | readSensor | サウンドセンサの値を取得してsensorに格納 | なし | int | loop |
| F2 | 拍手検出 | detectClap | sensorとlastSoundの差から拍手を検出 | sensor | bool | loop |
| F3 | 拍手カウント管理 | updateClap | clapCountとfirstClapTimeを更新 | 検出結果 | なし | loop |
| F4 | 拍手判定 | processClap | 2回・3回判定して処理を決定 | なし | なし | loop |
| F5 | 再生状態切替 | togglePlaying | playingのON/OFF切替 | なし | なし | F4内 |
| F6 | 曲変更 | changeMusic | sound（曲番号）を変更しnoteIndexをリセット | なし | なし | F4内 |
| F7 | 音楽再生 | playMusic | playingがtrueなら音を再生 | なし | なし | loop |
| F8 | 音更新 | updateNote | noteIndexとlastNoteTimeで次の音へ進む | なし | なし | F7内 |
| F9 | 入力無効判定 |isInputBlocked | lastTriggerTimeから1秒以内か確認 | なし | bool | loop |

> loop() の中で全部書こうとしていませんか？
> 機能ごとに関数を分けることで、グループレビューや単体テストがしやすくなります。

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| センサ読み取り | 毎ループ（数ms） | millis | 音の変化は瞬間的なので常に監視が必要 |
| 拍手検出 | 毎ループ | millis | 拍手は短時間のピークのため取りこぼし防止 |
| 拍手カウント管理 | 毎ループ | millis | 回数と時間をリアルタイムで管理する必要がある |
| 拍手判定（2回/3回） | 約0.5秒ごと | millis | 一定時間待って回数を確定するため |
| 入力無効判定 | 毎ループ | millis | 誤作動防止のため常にチェックする必要がある |
| 音楽再生（1音） | 約200〜500ms | millis | 音の長さに応じて切り替えるため |
| 音の進行（次の音） | 約200〜500ms | millis | 曲のテンポを維持するため |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。
